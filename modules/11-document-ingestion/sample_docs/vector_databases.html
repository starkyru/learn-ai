Vector Databases

What is a vector database?

A vector database stores high-dimensional numerical vectors alongside metadata and supports fast approximate nearest-neighbour (ANN) search. Instead of filtering rows by exact value, you query by semantic similarity: "give me the 10 vectors most similar to this query vector."

The core data structure is usually an HNSW (Hierarchical Navigable Small World) graph, which reduces search from O(n) to O(log n) at the cost of a small probability of missing the true nearest neighbour.

Popular options

  • Chroma — lightweight, in-process, great for prototyping; used in this course.
  • Qdrant — production-grade, Rust-based, supports filtering and payload storage.
  • Pinecone — managed cloud service; no infrastructure to run.
  • pgvector — a PostgreSQL extension; keeps vectors inside your existing relational DB.
  • Weaviate — GraphQL API, built-in modules for cross-encoders and summarisation.

When to use a vector DB

Use a vector database when your retrieval question is semantic rather than exact. Good fits:

  • RAG pipelines — retrieving relevant document chunks for LLM context.
  • Semantic search — "find products similar to this description."
  • Recommendations — "find items similar to what this user liked."
  • Anomaly detection — "find log entries far from all known clusters."
  • Deduplication — "find near-duplicate records."

For exact-match lookups (by ID, date range, category), a regular relational or key-value database is still the right tool.

Metadata filtering

Most vector databases support combining ANN search with metadata filters. Example: "find the 5 chunks most similar to this query, but only from documents published after 2024 and tagged 'security'."

This pre-filter (or post-filter) approach keeps retrieval focused and reduces the chance of serving stale or irrelevant context to the LLM.
